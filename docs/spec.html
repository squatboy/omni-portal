Omni Portal
VM, Kubernetes 클러스터, 개발 도구(ArgoCD, GitLab, Nexus)의 상태를 단일 화면에서 통합 모니터링하는 내부 인프라 포털. 외부 API 지연을 사용자 경로에서 격리하고, 최소 리소스로 높은 응답성을 유지.
● Go 1.25 Backend ● Next.js 16 Frontend ● PostgreSQL ● In-Memory Cache ● Docker Compose
Principle 1
30초 백그라운드 폴링
외부 API 지연 격리. 사용자는 캐시에서 즉시 응답.
Principle 2
AES-256-GCM 암호화
DB에 평문 없음. 수집 직전 메모리에서만 복호화.
Principle 3
Collector ↔ API 분리
수집 엔진과 API 서버는 독립. 인메모리 캐시로만 공유.
Principle 4
표준 상태 모델
K8s/ArgoCD/GitLab 상태값을 정규화. 일관된 UI.

데이터 수집 (Collector)
30초 주기 백그라운드 수집 엔진

⚙️ 구현 방식

  • Go Concurrency — Goroutine + Channel로 모든 연동 대상 병렬 수집. 전체 수집 시간 최소화.
  • RWMutex 캐시 — 수집 결과를 읽기/쓰기 락 보호 인메모리 스냅샷으로 관리. DB 미조회.
  • 장애 격리 — 특정 소스 실패 시 해당 소스만 down/unknown 처리. 전체 수집 중단 없음.
  • 크레덴셜 복호화 — 암호화된 Token은 수집 직전 서버 메모리에서만 복호화하여 HTTPS 헤더에 주입.
  • Timeout 처리 — CollectorTimeoutError로 개별 수집 타임아웃 격리.

📦 스냅샷 HTTP 상태 코드

  • 200 OK — 모든 source 정상 또는 모두 unknown(error 없음, 초기 상태)
  • 207 Multi-Status — 일부 runtime source 실패
  • 502 Bad Gateway — vm/kubernetes/argocd/gitlab/nexus 전체 실패

🔗 Source Envelope 상태 코드

  • 200 OK — 정상 또는 미판단 상태
  • 403 — 권한 오류
  • 422 — 입력성 upstream 오류
  • 503 — rate limit
  • 504 — timeout
  • 502 — 연결 실패 / upstream 5xx / parse 오류

IPAM
IPv4 주소 자원 관리 + ICMP 스캔
ℹ️ 계층 구조: Location → Network → Subnet → IP Address. 데이터는 PostgreSQL에 저장.

🗂️ 데이터 모델

  • Location — 최상위 논리 구역 (e.g., 데이터센터)
  • Network — Location 내부 논리 그룹 (CIDR 소유자는 Subnet)
  • Subnet — IPv4 CIDR, /24보다 큰 범위 거부, 같은 Location 내 overlap 불허
  • IP Address — Subnet 생성 시 사용 가능한 host IP row 자동 생성

📡 스캔 동작 및 예외 처리

  • Worker Pool — 크기 고정 64
  • used — ping 성공, consecutiveFailures = 0
  • offline — 과거 성공 이력 있는 IP에서 연속 3회 실패
  • free — 성공 이력 없는 IP는 실패해도 유지
  • Auto Discovery — interval: 1800/3600/14400/43200/86400초, 기본 3600초
  • Scheduler — dashboard collect runner와 별도로 독립 실행
  • 강제 재스캔 우회 — 스캔 비정상 중단 상황에 대비하여, 스캔 시작 후 15분이 경과한 Subnet은 중복 실행 차단(409 Conflict)을 우회하고 강제 재스캔 허용

🔒 데이터 정합성 제약

  • CIDR 제약 — IPv4 전용. /24보다 큰 범위(예: /23, /16)는 등록 거부
  • 중복 방지 — 동일 Location 내에서는 Network가 달라도 Subnet CIDR 겹침 불허
  • CIDR 불변 — Subnet CIDR은 생성 후 수정 불가. 변경 시 삭제 후 재생성 필요
  • 자동 IP 생성 — Subnet 생성 시 CIDR 범위 내 모든 host IP row가 자동 삽입됨
  • 연쇄 삭제 — Location / Network / Subnet 삭제 시 PostgreSQL FK Cascade로 하위 데이터 함께 물리 삭제
  • 주기 제한 — Auto Discovery 주기는 1800 / 3600 / 14400 / 43200 / 86400초 중 하나만 허용. 기본값 3600초

⚠️ 예외 처리 및 동시성 제어

  • Ping 타임아웃 — Go context timeout 2초 + OS ping 옵션으로 응답 대기 1초 이중 적용. 초과 시 해당 IP를 실패로 처리
  • 동시 스캔 차단 — 이미 실행 중인 Subnet에 중복 스캔 요청이 들어오면 409 Conflict를 반환하고 즉시 종료
  • 강제 재스캔 (Stale Lease 우회) — 스캔 시작 후 15분이 경과한 Subnet은 비정상 중단으로 간주하여 409 차단을 우회하고 재스캔 허용
  • 결과 덮어쓰기 방지 — 완료 반영 시 subnetId + startedAt lease를 검증. 지연된 이전 스캔(A)의 결과가 더 나중에 도착하더라도 최신 스캔(B) 반영 이후엔 A 결과를 무시하고 스킵하여 상태 꼬임 방지

📜 스캔 히스토리 (Scan History)

  • 이력 기록 — 스캔 완료 시 요약(ipam_scan_history)과 상태 변경 내역(ipam_scan_history_changes)을 단일 트랜잭션으로 커밋
  • 트랜잭션 바운더리 — 스캔 완료 시 (IP bulk update, Subnet status 갱신, history summary/change row 기록, retention pruning)을 함께 커밋하며, 스캔 실패 시에는 (Subnet failed status 반영, 실패 history 기록, retention pruning)을 함께 커밋
  • 실패 이력 — 실패한 스캔의 경우, 에러 메시지(error)와 발생 시각만 기록하며 count 및 변경 내역(diff)은 비워둠
  • 보존 정책 (Retention) — Subnet별로 최근 20개의 히스토리 행만 유지하며, 새로운 이력 추가 시 초과분은 데이터베이스에서 물리 삭제(Hard Delete)
  • 효율적 저장 — 데이터 효율성을 위해 전체 IP 주소 스냅샷은 보관하지 않고 변경된 주소(Diff)만 기록

🔄 ScanSubnet() 실행 플로우 goroutine postgres icmp

1
MarkIPAMScanStarted DB write
스캔 시작 시각(startedAt)을 DB에 기록. 이미 스캔 중인 경우 ErrConflict(409)를 반환해 동일 Subnet 중복 실행 방지 (시작 15분 경과 시 예외적으로 차단 우회).
2
ListIPAMScanAddresses DB read
Subnet의 모든 IP 레코드 로드. 각 레코드: status, consecutiveFailures, lastSeenAt 포함. 실패 시 markFailed 후 종료.
3
scanAddresses() — 64 Worker Goroutine 병렬 ICMP 64 goroutines ICMP
Producer goroutine이 jobs channel에 IP 주소 순차 전송. 64개 Worker가 receive하여 ICMP ping 수행. 결과는 버퍼드 results channel으로 전송. WaitGroup으로 전체 완료 후 results 채널 close.
4
classifyScanResult() — 상태 분류
메인 goroutine이 results를 receive하며 분류. ping 결과 + hadSuccess(과거 이력) 조합으로 used / offline / free 결정. 아래 시뮬레이터에서 직접 확인 가능.
5
BulkApplyIPAMScanResults DB bulk write
전체 IP 결과, Subnet 상태, Scan History 등을 단일 트랜잭션으로 반영. subnetId + startedAt lease 검증을 거쳐, 지연 완료된 이전 스캔 결과가 최신 결과를 덮어쓰는 상태 꼬임 방지.
6
IPAMScanSummary 반환
{ Total, Used, Offline, Free } 집계 반환. API 응답 및 스케줄러 로그에 활용.

⏰ IPAM Scheduler — Dashboard Runner와 독립 실행

⚠️ Scheduler는 Dashboard Collector Runner와 완전히 별개의 Goroutine에서 실행. 서로 간섭 없음.

📋 스케줄러 로직

  • 시작 즉시 1회 실행 — Start(ctx) 호출 시 scanDue(ctx) 즉시 실행
  • 1분 ticker — 이후 1분마다 ScanExecutor.ScanDue 호출
  • ScanDue 캡슐화 — 만료 subnet 조회(ListDueIPAMSubnets), 스캔 락(claim) 및 Skip 처리가 ScanDue 내부 계약으로 완전 캡슐화되어 진행
  • ListDueIPAMSubnets(limit=10) — Auto Discovery interval 만료 subnet 최대 10개 조회
  • ErrConflict 무시 — 이미 스캔 중인 subnet 건너뜀
  • Context 취소 감지 — ctx.Err()로 graceful shutdown

Login / Setup — DB에 사용자 없을 때만 Setup 화면 허용. 이후 공개 회원가입 차단. 로그인 성공/실패 동일 포맷 응답(계정 존재 유무 비노출). 과도한 실패 시 일시 차단.
bcrypt 검증 — 비밀번호는 bcrypt 단방향 해시로 저장. 무차별 대입 완화. 신규 사용자 첫 로그인 후 즉시 비밀번호 변경 강제.
Opaque Session Token — 난수 기반 토큰을 생성. DB에 토큰 해시 + 만료 시각 저장. JWT와 달리 서버 측에서 수명과 권한을 완전 제어. Logout 시 즉시 revoke 가능.
쿠키 보안 — HttpOnly: JS에서 쿠키 접근 불가(XSS 방어). SameSite=Lax: 외부 사이트 요청 시 쿠키 미전송(CSRF 방어).
RBAC (admin/viewer) — Gin 미들웨어에서 역할 검증. Admin: 모든 API. Viewer: 대시보드+IPAM 조회만. Dev 환경에서만 NEXT_PUBLIC_OMNI_MOCK=true로 인증 우회 가능.
Admin
전체 접근
대시보드 + IPAM + Manage(VM/Integration/User CRUD) + 모든 API
Viewer
조회 전용
대시보드 + IPAM 조회만. Manage 및 mutation API 완전 차단.

🌐 IPAM 도메인 (PostgreSQL)

계층 구조: Location → Network → Subnet → Address. 모두 PostgreSQL에 영구 저장. ID는 각 계층 접두사(loc_, net_, subnet_, ip_) + UUID.
IPAMLocation
필드 타입 설명
id string loc_ 접두사 UUID
name string 구역명 (e.g. 서울 IDC)
description string? 선택 설명
createdAt / updatedAt string (RFC3339) 생성·수정 시각
IPAMNetwork
필드 타입 설명
id string net_ 접두사 UUID
locationId string 상위 Location FK — 불변
name string 네트워크 그룹명
description string? 선택 설명
IPAMSubnet
필드 타입 설명
id string subnet_ 접두사 UUID
networkId string 상위 Network FK — 불변
cidr string IPv4 CIDR · /24 이상만 허용 · 불변 · 동일 Location 내 overlap 불허
autoDiscovery bool 자동 스캔 활성화 여부
scanIntervalSeconds int 스캔 주기 (1800 / 3600 / 14400 / 43200 / 86400) · 기본 3600
lastScanStartedAt string? 마지막 스캔 시작 시각
lastScanCompletedAt string? 마지막 스캔 완료 시각
lastScanStatus string? running / completed / failed
lastScanError string? 실패 사유 메시지 (최대 500자)
IPAMAddress
필드 타입 설명
id string ip_ 접두사 UUID
subnetId string 상위 Subnet FK
address string IPv4 주소 — Subnet 생성 시 host IP 전체 자동 생성
status enum used / offline / free
hostname string? admin이 수동 입력
description string? admin이 수동 입력
lastScannedAt string? 마지막 ping 시도 시각
lastSeenAt string? 마지막 ping 성공 시각 · null이면 성공 이력 없음(free)
consecutiveFailures int 연속 ping 실패 횟수 · ≥3 이면 offline 전이 조건 충족
IPAMScanSummary
필드 타입 설명
subnetId string 대상 Subnet ID
total / used / offline / free int 스캔 직후 상태별 IP 주소 수 집계
startedAt / completedAt string 스캔 시작·완료 시각
subnet IPAMSubnet 대상 Subnet의 상세 메타데이터
IPAMScanHistory
필드 타입 설명
id string UUID
subnetId string 대상 Subnet FK
subnetName / subnetCidr string 스캔 당시 Subnet 명칭 및 CIDR 스냅샷 복사
startedAt string? 스캔 시작 시각 (실패 시 null 가능)
completedAt string 스캔 완료 시각
status enum completed / failed
total / used / offline / free int? 완료 상태의 경우 각 IP 수 집계 (실패 시 null)
error string? 스캔 실패 시 에러 메시지 (성공 시 null)
IPAMScanHistoryChange
필드 타입 설명
id string UUID
historyId string 상위 IPAMScanHistory FK
address string 상태 변경이 일어난 IP 주소
previousStatus / currentStatus enum 변경 전/후 IP 상태 (used / offline / free)
previousLastSeenAt / currentLastSeenAt string? 변경 전/후 마지막 ping 성공 시각 스냅샷
previousConsecutiveFailures / currentConsecutiveFailures int 변경 전/후 연속 ping 실패 횟수

📊 대시보드 수집 도메인 (In-Memory)

Collector Runner가 주기적으로 수집하여 In-Memory Cache에 기록. 프로세스 재시작 시 초기화되고 첫 수집 사이클에서 재생성됨.
CollectEnvelope<T>
각 소스 수집 결과를 감싸는 제네릭 래퍼. DashboardSnapshot의 모든 필드가 이 타입.
필드 타입 설명
source string overview / vms / kubernetes / argocd / gitlab / nexus
status enum ok / progressing / down / timeout / permission_error / stale / unknown
attemptedAt string 수집 시도 시각
collectedAt string? 마지막 성공 수집 시각 — 실패 시 null
stale bool 마지막 성공 후 5분 경과 시 true
error CollectError? code (TIMEOUT / PERMISSION_DENIED / CONNECTION_FAILED / UNKNOWN_ERROR) · message · upstreamStatus
data T 소스별 실제 수집 데이터
DashboardSnapshot — 소스별 데이터(T) 요약
필드 (source) 데이터 타입 T 주요 내용
overview OverviewData health (ok / degraded / unknown) + 소스별 SourceSummary 목록
vms VmsData VmStatus 목록 — id, name, address, state (up / down / unknown), lastCheckedAt
kubernetes KubernetesData Nodes (CPU·Memory %), Workloads (deploy/sts/ds), Pods, Services, Ingresses, PVCs
argocd ArgoCdData Application 목록 — syncStatus (Synced/OutOfSync), healthStatus (Healthy/Degraded/…), revision
gitlab GitLabData Project 목록 — latestCommit (sha, title, author), latestPipeline (status, ref, link)
nexus NexusData reachable, httpStatus, checkedAt

🔐 사용자 도메인 (PostgreSQL)

필드 타입 설명
id string UUID
username string 로그인 ID (unique)
role enum admin — 전체 접근 / viewer — 조회 전용

⚙️ 스캔 내부 모델 (In-Process)

DB 왕복 최소화를 위한 경량 타입. API 응답으로 노출되지 않음.
IPAMScanAddress
스캔 시작 시 DB에서 일괄 조회하는 경량 뷰
id, subnetId, address
status, lastSeenAt, consecutiveFailures
IPAMScanResult
Worker 결과 채널에서 취합한 중간 타입
addressId, status
lastScannedAt, lastSeenAt, consecutiveFailures
→ BulkApplyIPAMScanResults()로 단일 트랜잭션 반영

AES-256-GCM 자격 증명
DB에 평문 없음 · 수집 직전 복호화
알고리즘: AES-256-GCM (인증 암호화)
키: OMNI_SECRET_KEY (32바이트)
AD(Additional Data): integration_type:integration_id:secret_name — 데이터 변조 탐지
복호화 시점: 수집 실행 직전 서버 메모리에서만. UI 조회 시 마스킹.
저장 형식: 암호화된 바이너리 blob을 PostgreSQL에 저장.
🎫
Opaque Session Token
JWT 없음 · 서버 측 완전 제어
생성: 서버에서 난수 기반 opaque 토큰 생성
저장: DB에 토큰 해시 + 만료 시각 저장 (평문 토큰 미저장)
전달: HttpOnly; SameSite=Lax 쿠키로 브라우저에 전달
검증: 매 요청마다 DB에서 해시 비교 + 만료 확인
Revoke: Logout 시 DB 레코드 즉시 비활성화 (JWT의 stateless 단점 없음)
🔑
bcrypt 비밀번호
단방향 해시 · 무차별 대입 완화
비밀번호는 bcrypt로 단방향 해시 저장. 복호화 불가.
신규 사용자는 첫 로그인 후 즉시 비밀번호 변경 강제.
로그인 실패 횟수 추적 → 과도한 접근 시 일시 차단.
성공/실패 동일 오류 포맷 → 계정 존재 유무 비노출.

배포

⚠️ Omni는 Kubernetes 클러스터 외부 VM에 배포. 클러스터 장애 시에도 Omni가 살아있어야 관측이 가능하기 때문.

📁 배포 번들 구조

/opt/omni-portal/deploy/
  docker-compose.yml
  .env                        # OMNI_VERSION, DB 설정, OMNI_SECRET_KEY
  certs/kubernetes-ca.crt     # K8s private CA 사용 시만

⚙️ 환경 변수

  • OMNI_VERSION — 릴리스 이미지 태그 (e.g., v1.0.1). latest 사용 금지.
  • DATABASE_URL — PostgreSQL 연결 문자열
  • POSTGRES_DB / USER / PASSWORD — Compose DB 설정
  • OMNI_SECRET_KEY — 32바이트 자격 증명 암호화 키
  • API_URL — http://backend:8080 (Compose 내부 통신)
